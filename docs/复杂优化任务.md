# 妙语成语 - 复杂优化任务

这些任务涉及产品体验、状态流转或多模块重构，建议先评审方案，再拆分实现。适合 1-2 人协作推进。

## 当前执行状态

已由 Codex 先落地一批基础能力，并通过 `hvigorw assembleApp --no-daemon` 构建验证：

- 图鉴主成长线：已升级收藏页为图鉴式列表，支持收集进度、分类/难度筛选、未收集占位和详情弹窗；搜索、分类进度详情仍可继续补。
- 错题复习闭环：已新增结构化错题服务，答错自动入库，错题页可标记“答对一次/仍不会”，连续答对后进入已掌握；复习页专门抽错题仍可继续补。
- 每日挑战打卡体系：已新增每日挑战记录服务，完成后写入日期记录，首页展示今日状态和连续天数；日历视图仍可继续补。
- 统一出题引擎：已新增 `QuestionService.ets`，经典模式和每日挑战共用判题、填空、提示、题型权重逻辑；复习模式接入可作为下一步。
- 分层提示系统：经典模式和每日挑战已优先基于题库拼音字段生成提示，避免汉字映射误差；不同题型的 UI 反馈还可继续细化。
- 新手难度保护：已在动态题型选择中加入前 3 局保护，新用户优先进入拼音/填空题型。
- 分享卡片：第一阶段已完成，`ShareService.ets` 可生成成绩分享文案并复制到剪贴板；图片卡片和系统分享面板仍待后续接入。

## 1. 成语图鉴主成长线

**用户价值：**
- 让用户有长期收集目标，而不是只玩一局小游戏。
- 把“答对成语”转化为“点亮图鉴”的成长反馈。

**目标体验：**
- 图鉴首页展示 `已收集 X/510`。
- 展示总进度、分类进度、难度进度。
- 已收集成语显示完整卡片。
- 未收集成语可显示灰色占位或只显示分类/难度。
- 成语详情包含成语、拼音、释义、出处、例句。

**建议设计：**
- `CollectionPage` 改为图鉴首页。
- 增加分类筛选、难度筛选、搜索。
- 答对后自动收集，并在游戏页/结果页给轻提示。
- 和成就系统联动，例如收集 10/50/200/510。

**涉及模块：**
- `pages/CollectionPage.ets`
- `service/CollectionService.ets`
- `service/ChengyuService.ets`
- `service/AchievementService.ets`
- `model/ChengyuModel.ets`

**关键数据：**
- `collected_chengyu`
- 成语总数来自 `getTotalCount()`
- 成语分类、难度来自 rawfile 数据。

**风险点：**
- 收藏服务依赖 `ChengyuService` 已加载题库，页面进入时要先确保数据已加载。
- 图鉴数据量 510 条，列表要用 List 而不是一次性复杂嵌套渲染。

**验收标准：**
- 答对一个新成语后，图鉴进度增加。
- 图鉴页能按分类/难度筛选。
- 未收集和已收集状态清晰。
- 构建通过。

## 2. 错题复习闭环

**用户价值：**
- 用户不只是知道自己错了，还能把错题练会。

**目标体验：**
- 答错成语进入错题本。
- 复习模式只出错题或未掌握成语。
- 连续答对 2-3 次后标记为“已掌握”。
- 错题页展示待复习、已掌握、累计错误次数。

**建议数据结构：**

```json
{
  "word": "画龙点睛",
  "wrongCount": 2,
  "reviewCorrectStreak": 1,
  "mastered": false,
  "lastWrongAt": "2026-05-12",
  "lastReviewedAt": "2026-05-12"
}
```

**建议实现：**
- 新增或扩展 `StorageService` 读写错题记录。
- `GameViewModel.submitAnswer()` 答错时记录错题。
- `ReviewPage` 或 `WrongBookPage` 提供复习入口。
- 复习答对更新 `reviewCorrectStreak`。
- 连续答对达到阈值后标记 mastered。

**涉及模块：**
- `pages/WrongBookPage.ets`
- `pages/ReviewPage.ets`
- `viewmodel/GameViewModel.ets`
- `service/StorageService.ets`
- 可新增：`service/WrongBookService.ets`

**风险点：**
- 不要只存 word 字符串，后续统计会不够用。
- 注意旧版本已有错题数据时的兼容。

**验收标准：**
- 答错后错题本出现该成语。
- 复习模式能只抽错题。
- 连续答对后错题从“待复习”变为“已掌握”。

## 3. 每日挑战打卡体系

**用户价值：**
- 提供每天打开一次的明确理由。

**当前状态：**
- 已实现日期 seed，同一天题目固定。
- 已有 `daily_done_date`，只能判断当天是否完成。

**目标体验：**
- 每天 10 题固定题。
- 完成后显示正确率、评级、连续挑战天数。
- 首页展示今日是否完成、连续天数。
- 日历视图展示本月完成情况。

**建议数据结构：**

```json
{
  "2026-05-12": {
    "completed": true,
    "correctCount": 8,
    "total": 10,
    "type": "explanation",
    "completedAt": "2026-05-12T12:00:00"
  }
}
```

**建议实现：**
- 新增 `DailyChallengeService.ets` 管理记录。
- 完成每日挑战时写入当天记录。
- 根据日期连续性计算 streak。
- `HomePage` 首页展示“今日挑战：未完成/已完成”。
- `DailyChallengePage` 完成页展示评级。

**涉及模块：**
- `pages/DailyChallengePage.ets`
- `pages/HomePage.ets`
- `service/StorageService.ets`
- 可新增：`service/DailyChallengeService.ets`
- `service/AchievementService.ets`

**风险点：**
- 日期字符串要统一 `YYYY-MM-DD`。
- 注意跨月连续天数计算。
- 不要依赖网络时间。

**验收标准：**
- 同一天重复进入题目不变。
- 完成后首页显示已完成。
- 连续完成多天后 streak 正确增加。

## 4. 统一出题引擎

**用户价值：**
- 各模式题型、提示、判题规则一致，减少“这个页面这样、那个页面那样”的割裂。

**当前问题：**
- 经典模式、每日挑战、复习模式各自生成题目和提示。
- 判题 normalize、填空生成、提示生成存在重复。

**目标设计：**
- 抽出统一题目模型和题目生成工具。
- 所有模式复用同一套判题和提示逻辑。

**建议新增：**
- `model/QuestionModel.ets`
- `service/QuestionService.ets`

**建议模型：**

```ts
interface Question {
  chengyu: Chengyu;
  originalIndex: number;
  type: string;
  prompt: string;
  hintText: string;
  answer: string;
}
```

**能力范围：**
- 根据题型生成 prompt。
- 根据 hintLevel 生成提示。
- 标准化答案并判题。
- 支持 seed 随机。
- 支持过滤已玩、错题、难度、分类。

**涉及模块：**
- `viewmodel/GameViewModel.ets`
- `pages/DailyChallengePage.ets`
- `pages/ReviewPage.ets`
- `service/ChengyuService.ets`
- `utils/PinyinUtil.ets`

**风险点：**
- 改动面较大，建议先新增服务并让每日挑战接入，再迁经典模式。
- 不要一次性重写所有页面。

**验收标准：**
- 同一题型在经典模式和每日挑战中提示一致。
- 判题忽略标点规则一致。
- 构建通过，核心模式可玩。

## 5. 分层提示系统

**用户价值：**
- 用户卡住时能得到帮助，但不会直接失去挑战感。

**目标体验：**
- 第 0 层：显示首字母。
- 第 1 层：显示一个汉字。
- 第 2 层：显示两个汉字或出处。
- 第 3 层：显示完整答案。
- 每次提示扣分，并在 UI 上明确反馈。

**建议设计：**
- 不同题型可以有不同提示策略。
- 拼音题提示释义。
- 释义题提示拼音首字母。
- 填空题提示出处或拼音。
- 接龙题提示首字母或释义。

**涉及模块：**
- `utils/PinyinUtil.ets`
- `viewmodel/GameViewModel.ets`
- `pages/GamePage.ets`
- `pages/DailyChallengePage.ets`
- 建议配合统一出题引擎做。

**风险点：**
- 当前提示语义在文档和代码中曾不一致，改之前要先定规则。

**验收标准：**
- 所有题型点击提示都有合理反馈。
- 扣分一致。
- 不会出现错误拼音首字母。

## 6. 新手难度保护

**用户价值：**
- 第一次打开不被难题劝退。

**目标体验：**
- 新用户默认简单题。
- 前几局优先四选一、填空、拼音题。
- 逐步开放释义题、接龙题、困难难度。

**建议实现：**
- 根据总场次、正确率、连续答对数计算题型权重。
- 新用户前 3 局固定较低难度。
- 首页默认选中简单或智能难度。

**涉及模块：**
- `viewmodel/GameViewModel.ets`
- `pages/HomePage.ets`
- `service/StorageService.ets`

**风险点：**
- 不要把老用户强制降难度。
- 动态难度要可解释，避免用户感觉随机。

**验收标准：**
- 新用户第一局不会进入过难题型。
- 老用户仍能选择困难和指定分类。

## 7. 分享卡片

**用户价值：**
- 提供传播和自我展示动机。

**目标体验：**
- 游戏结束生成成绩卡片。
- 每日挑战完成生成挑战卡片。
- 成语详情可生成“今天学会一个成语”卡片。

**建议实现阶段：**
- 第一阶段：复制分享文案。
- 第二阶段：生成本地图片卡片。
- 第三阶段：接系统分享。

**涉及模块：**
- `pages/GameOverPage.ets`
- `pages/DailyChallengePage.ets`
- `pages/CollectionPage.ets`
- `service/ShareService.ets`

**风险点：**
- HarmonyOS 分享 API 当前项目里是占位，需按目标 SDK 验证。
- 图片生成和保存相册可能涉及权限。

**验收标准：**
- 至少能复制分享文案。
- 不依赖网络。
